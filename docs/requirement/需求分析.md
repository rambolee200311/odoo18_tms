# 运输物流管理系统 — 需求分析

## 1. 业务领域

面向**欧洲第三方物流（3PL）服务商**的陆运运输管理系统，覆盖从提货需求到运输结算的完整生命周期。

### 核心业务实体

- **Terminal（码头/货站）**：海运码头、空运货站、集装箱堆场（CY），作为运输起点
- **Warehouse（仓库）**：我方运营的仓储设施，分布在鹿特丹、海牙、安特卫普等多地
- **Customer（客户）**：委托我方安排运输的客户
- **Carrier（卡车公司/承运商）**：执行运输的外部车队
- **Container（集装箱）**：运输的基本单元，按柜管理
- **Pallet/Piece（托/件）**：非集装箱货物，按托/件管理

## 2. 覆盖场景（共 8 项）

### 场景 1：Terminal → 我方仓库

| 属性 | 说明 |
|------|------|
| 货物形态 | 大部分为集装箱，少量为托/件（分开建需求） |
| 流程类型 | **计划驱动型**（无对外报价） |
| 流程链路 | 需求 → 计划（排期日历）→ 运输订单 |
| 系统路径 | `transport.request` (plan_driven) → Schedule Calendar → `pickup.plan` → `tlmp.transport.order` |
| 说明 | 货物到港/到站后，提柜/提货回仓，走内部调度，默认卡车公司 |

### 场景 2：Terminal → 客户地址

| 属性 | 说明 |
|------|------|
| 货物形态 | 大部分为集装箱，少量为托/件 |
| 流程类型 | **商务报价型**（对外接单计费） |
| 流程链路 | 需求 → 运输请求 → 询价 → 报价 → 订单 |
| 系统路径 | `transport.request` (commercial) → `tlmp.transport.inquiry` → `tlmp.transport.quote` → `tlmp.transport.order` |
| 说明 | 需向多家卡车公司询价比价，我方加价后向客户报价 |

### 场景 3：我方仓库 → 客户地址

| 属性 | 说明 |
|------|------|
| 货物形态 | 大部分为托/件（仓库出库配送） |
| 配送方式 | 卡车公司或快递 |
| 流程类型 | **商务报价型** |
| 流程链路 | 需求 → 询价 → 报价 → 订单 |
| 说明 | 客户委托配送，需先询价报价，确认后才执行出库 |

### 场景 4：客户地址 A → 客户地址 B

| 属性 | 说明 |
|------|------|
| 货物形态 | 大部分为托/件 |
| 配送方式 | 卡车公司或快递 |
| 流程类型 | **商务报价型** |
| 说明 | 类似场景 3，起点终点均为客户指定地址 |

### 场景 5：我方仓库 A → 我方仓库 B（调拨）

| 属性 | 说明 |
|------|------|
| 货物形态 | 柜或托/件 |
| 流程类型 | **计划驱动型** |
| 流程链路 | 需求 → 计划（排期日历）→ 运输订单 |
| 系统路径 | `transport.request` (plan_driven, warehouse_transfer) → Schedule → `pickup.plan` → `tlmp.transport.order` |
| 说明 | 内部调拨，无客户报价环节，同场景 1 流程 |

### 场景 6：客户地址 → 我方仓库（退货回仓）

| 属性 | 说明 |
|------|------|
| 货物形态 | 托/件 |
| 配送方式 | 卡车公司 |
| 流程类型 | **商务报价型** |
| 流程链路 | 需求 → 询价 → 报价 → 订单 |
| 说明 | 仅适用于由我司承接运输、上门提货的退货单。客户自行安排送货到仓的不归 TMS 管控 |
| 运输类型 | `reverse_logistics` |

### 场景 7：柜到仓后换柜

| 属性 | 说明 |
|------|------|
| 范围 | 运输订单内每条柜的附属标记 |
| 变更 | `tlmp.transport.container` 新增两个字段 |
| 说明 | 集装箱运到我方仓库后，如需换柜到船公司指定地点，在容器明细中标记"需换柜 + 换柜地点" |

### 场景 8：Depot / Terminal ↔ 我方仓库（空柜调拨）

| 属性 | 说明 |
|------|------|
| 货物形态 | 空集装箱 |
| 流程类型 | **计划驱动型** |
| 流程链路 | 柜务申请 → 计划 → 调柜订单 |
| 系统路径 | `container.service.request` → `tlmp.transport.order` |
| 方向 | 双向：堆场→仓库 或 仓库→堆场 |
| 说明 | 独立于货载运输的纯空柜移动管理 |

## 3. 流程二分法

```
                           tlmp.transport.request
                                    │
                              request_type?
                                    │
         ┌──────────────────────────┴──────────────────────────┐
         ▼                                                      ▼
   计划驱动型（内部调度）                              商务报价型（对外接单）
         │                                                      │
         ▼                                                      ▼
   场景 1: Terminal → 仓库                         场景 2: Terminal → 客户
   场景 5: 仓库A → 仓库B                           场景 3: 仓库 → 客户
   场景 8: Depot ↔ 仓库（空柜）                     场景 4: 客户A → 客户B
                                                    场景 6: 客户 → 仓库（退货）
         │                                                      │
         ▼                                                      ▼
   Schedule Calendar (排期日历）                     tlmp.transport.inquiry
   （创建 pickup.plan）                                   （向承运商询价）
         │                                           tlmp.transport.inquiry
         ▼                                           （向承运商询价）
   tlmp.transport.order                                    │
   （直接派车）                                      tlmp.transport.quote
                                                   （我方加价 → 客户确认）
                                                           │
                                                           ▼
                                                    tlmp.transport.order
                                                    （确认后自动创建）
```

## 4. 数据模型

### 4.1 `pickup.plan`（提货需求）

计划链路排期子单据（非顶层入口），支持柜和托件两种货物类型。Sprint3 起入口统一为 transport.request。

| 字段 | 类型 | 说明 |
|------|------|------|
| `name` | Char | 需求编号，自动序列 |
| `cargo_type` | Selection | container / pallet，决定显示哪些明细字段 |
| `source_type` | Selection | iff / manual，需求来源 |
| `iff_requirement_ref` | Reference | 可选，指向 wd_iffm 的 `import.pickup.requirement` |
| `destination_type` | Selection | warehouse / warehouse_transfer / customer / self_pickup |
| `warehouse_id` | Many2one | 目标仓库 |
| `source_warehouse_id` | Many2one | 调拨场景的发货仓库 |
| `delivery_address/contact/phone` | Char/Text | 客户地址信息 |
| `partner_id` | Many2one | 客户（非仓库流程必填） |
| `pallet_count / package_count / cargo_weight / cargo_volume / cargo_description` | 各种 | 托件货物明细，cargo_type=pallet 时显示 |
| `container_line_ids` | One2many | 集装箱行明细，cargo_type=container 时显示 |
| `carrier_id` | Many2one | 卡车公司 |
| `planned_pickup_date` | Datetime | 计划提货时间 |
| `driver_name / driver_phone / vehicle_plate` | Char | 司机信息 |
| `schedule_ids` | One2many | 关联的排期记录 |
| `transport_request_id` | Many2one | 关联的运输请求（商务报价型流程） |
| `inquiry_id` | Many2one | 关联的询价单 |
| `transport_order_id` | Many2one | 关联的运输订单 |
| `notes` | Text | 备注 |

### 4.2 `pickup.plan.container.line`（集装箱行）

| 字段 | 说明 |
|------|------|
| `plan_id` | 所属需求 |
| `container_number` | 柜号 |
| `container_type` | 箱型（20GP/40GP/40HQ/40HC/45HQ/OT/FR/RF） |
| `weight` | 重量 (kg) |
| `bl_number` | 提单号 |
| `seal_number` | 封条号 |
| `notes` | 备注 |

### 4.3 `pickup.plan.schedule`（排期记录）

| 字段 | 说明 |
|------|------|
| `container_line_id` | 关联集装箱行（可选，柜类型用） |
| `plan_id` | 关联需求（直连字段，托件类型用此关联） |
| `scheduled_date` | 排期日期 |
| `container_number` | 柜号（related） |
| `item_type` | 计算字段：container / pallet |
| `display_name` | 计算字段：展示名称 |

### 4.4 `container.service.request`（空柜调拨申请）

| 字段 | 类型 | 说明 |
|------|------|------|
| `name` | Char | 申请编号，自动序列 |
| `request_type` | Selection | depot_to_warehouse / warehouse_to_depot |
| `container_number` | Char | 柜号 |
| `container_type` | Selection | 箱型 |
| `quantity` | Integer | 数量 |
| `depot_location_id` | Many2one | 堆场/码头 |
| `warehouse_id` | Many2one | 仓库 |
| `carrier_id` | Many2one | 卡车公司 |
| `planned_date` | Date | 计划日期 |
| `state` | Selection | draft / confirmed / completed / cancelled |
| `transport_order_id` | Many2one | 关联调柜订单 |

### 4.5 `tlmp.transport.container`（运输订单容器）（增量）

| 新增字段 | 说明 |
|---------|------|
| `needs_swap` | 布尔，是否需换柜 |
| `swap_location_id` | Many2one，换柜地点 |

## 5. 用户界面

### 5.1 提货需求表单（`pickup.plan`）

- 顶部根据 `destination_type` 显示不同操作按钮
  - 仓库 / 调拨 → **"Go to Schedule"** → 跳转排期日历
  - 客户 / 自提 → **"Create Transport Request"** → 创建运输请求，走询价报价
- `cargo_type = container` 时显示集装箱行可编辑列表
- `cargo_type = pallet` 时显示托/件字段
- `destination_type = warehouse_transfer` 时显示发/收两个仓库选择

### 5.2 排期日历（Schedule）

- 左侧面板：可拖拽的集装箱/托件列表
  - 目的地过滤（All / Warehouse / Customer / Self-Pickup）
  - 仓库过滤（目的地为 Warehouse 时展开）
  - 每个卡片显示柜号 + 目的地标签 + 仓库名
  - 托件卡片黄色左边框，显示货描 + 托数 + 重量
- 右侧日历：按月网格
  - 每月导航（前后切换），当天高亮
  - 每格显示日期 + 当日柜/托量
  - 已排期的柜/托显示在格内，可点 x 移除
- 交互：HTML5 Drag & Drop
  - 左→右：排期到某天
  - 右→左：取消排期
  - 日历格内点击 x：取消排期

### 5.3 空柜调拨表单（`container.service.request`）

- 状态栏：draft → confirmed → completed
- confirmed 状态显示 **"Create Dispatch Order"** 按钮
- 双向选择：Depot → Warehouse 或 Warehouse → Depot
- 调度信息：卡车公司 + 计划日期 + 司机信息

### 5.4 运输订单容器明细

- 列表视图新增 `needs_swap` 和 `swap_location_id` 列
- 表单视图新增"Swap / Return"分组

## 7. 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 流程二分 | 计划驱动型 vs 商务报价型 | 内部调度不需询价，对外接单必需计价 |
| Schedule 使用范围 | 仅港到仓和调拨 | 其他场景无固定日期排期需求 |
| 柜/托混合 | 不混合，分开建需求 | 数据结构差异大，混合增加复杂度 |
| 空柜调拨 vs 货载运输 | 独立模型 `container.service.request` | 无客户参与、无报价环节、双向可逆 |
| 换柜标记 | 容器级字段，非独立流程 | 无需新增菜单和审批流 |
| Terminal vs Port | 使用 Terminal | 覆盖海运码头、空运货站、集装箱堆场 |
| 跨模块引用（iff） | Reference 字段，无硬依赖 | 避免 `transport_logistics_management` 依赖 `wd_iffm` |
| 默认卡车公司 | 系统参数 `tlmp.default_pickup_carrier_id` | 灵活可配，用户可在表单中修改 |
| 前端架构 | Odoo QWeb + Vanilla JS，无框架 | 轻量、无构建步骤、与 Odoo 深度集成 |
| 单据来源权限控制 | IFFM 导入的提货需求明细字段全部只读 | 仅手动创建单据支持编辑，防止上游数据被篡改 |
| 报价单回退机制 | Quote 驳回后可退回 Inquiry 状态 | 支持重新向多家承运商比价，无需删单重建 |
| POD 强约束 | 无有效签收 POD 的运输订单无法标记完成 | 不可生成结算账单，CMR/POD 是欧盟陆运法定对账依据 |
| 柜号主数据 | 新增 `container.master` 全局柜档案 | 所有柜明细外键绑定唯一柜号，支撑后续柜全生命周期 |
| 保税调拨标识 | `pickup.plan` 增加 `is_bonded_transfer` 布尔字段 | 勾选后触发保税系统账册核销信号 |
| 结算锁定规则 | 存在未完结事故单的运单禁止进入应收应付对账 | 从源头管住赔付漏记 |
| 保税调拨强制勾选 | 调拨出/入仓任意一方为保税仓时，`is_bonded_transfer` 强制必选 | 不可手动留空，确保保税账册核销不漏 |
| IFFM 来源生命周期锁定 | 上游 IFFM 单据作废/撤销时，TMS `pickup.plan` 自动置为只读 | 不可继续推进流程 |

## 8. 跨模块联动边界

### 8.1 与 IFFM 进口货代模块

| 方向 | 规则 |
|------|------|
| 数据流 | TMS 只读引用，仅拉取 IFFM 的提货需求数据 |
| 写权限 | **绝不反向修改 IFFM 任何单据字段**，提货需求仅作为下游执行环节 |
| 状态管控 | 上游单据状态由 IFFM 独立管控，TMS 不干预 |
| 明细保护 | IFFM 来源的 `pickup.plan` 所有明细字段只读，仅可推进流程 |

### 8.2 与 wd_bonded_wms 保税仓模块

| 方向 | 规则 |
|------|------|
| 数据流 | TMS 仅推送「调拨/入仓/出库」单据信号 |
| 库存归属 | 库存增减、账册额度、B3/T1 申报、核注核销 **全部由 WMS 模块执行** |
| 数据持有 | TMS 不持有任何保税库存数据，只做运输轨迹留痕 |
| 联动触发 | `pickup.plan.is_bonded_transfer = True` 时触发保税核销流程 |

## 10. 二期规划

### 10.1 事故与争议管理（Accident & Incident Management）— P0

现有仅 POD `state=disputed` + 文本备注，无法支撑实际业务需要。

**需求：**
- 独立的事故/事件报告模型 `transport.incident`
- 关联运输订单、集装箱、司机、车辆
- 事故类型枚举：货损、短少、延误、海关扣货、封志异常、拒收、交通事故
- 责任方枚举：承运商、我方操作、客户方、海关不可抗力
- 结构化字段：发生时间、地点、货物状态、照片证据、内部调查结论
- 理赔跟踪：保险单号、报案号、赔付金额、理赔状态
- **强规则**：存在未完结事故单的运输单，禁止生成客户账单、禁止承运商结算

**优先级：P0 — 二期首个迭代**

### 10.2 承运商档案与风控（Carrier Management）— P1

当前仅通过 `res.partner.is_carrier` 标记承运商身份，缺少准入与风控。

**需求：**
- 承运商资质：营业执照、运输许可证、保险凭证（含有效期）
- 保险管理：货险保单号、保额、有效期，到期预警
- 考核指标：月度货损率、准点率、投诉次数
- 黑名单：不良承运商拦截派单
- 逾期预警：POD 长期未回传、费用结算逾期

**优先级：P1 — 风控刚需，复杂度低于司机端**

### 10.3 司机端响应式门户（Driver Portal）— P2

**不做原生 APP**，使用 Odoo Portal 做手机适配页面。

**需求：**
- 查看今日任务列表（运输订单）
- 上报追踪事件：提货、发车、到达、卸货、异常
- 拍照上传：POD 签收照片、货损照片
- 电子签名
- 导航跳转外部地图链接（Google Maps / Waze），不集成 SDK

**优先级：P2 — 降低部署运维成本**

### 10.4 自动对账与结算（Automated Settlement）— P3

现有结算需人工比对账单和运输订单。

**需求：**
- 承运商对账单自动生成：按合同价 / 询价结果比对实际费用
- 差异标记：价差、附加费缺失、多收
- 客户账单自动生成：按报价 + 实际签收数据
- 与 `account.move` 的自动过账
- 结算周期管理：周结 / 月结 / 票结

**优先级：P3**

### 10.5 第三方 GPS 硬件对接 — P4

**作为可选增值插件，不纳入主干版本**。车队硬件选型不统一，强耦合第三方 API 会导致后续对接多厂商大量改代码。

**优先级：P4 — 永久可选插件**

### 二期优先级总表

| 项目 | 优先级 | 关联一期 | 预计复杂度 |
|------|--------|---------|-----------|
| 事故与争议管理 | **P0** | 结算锁定、POD | 中 |
| 承运商档案与风控 | **P1** | 运输订单派单 | 中 |
| 司机端响应式门户 | **P2** | 追踪、POD、CMR | 中 |
| 自动对账结算 | **P3** | 账单、结算 | 中 |
| 第三方 GPS 硬件 | **P4** | 追踪 | 高（硬件选型风险） |
