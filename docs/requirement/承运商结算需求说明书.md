# 1 总体评审结论与定位
## 1.1 总体结论
V1.2已完成全部P0架构整改，从简单账单核销升级为**企业级欧洲标准承运商批量结算系统**，对齐SAP TM、海外成熟物流TMS结算设计，完全适配Odoo数据模型规范，可直接作为Sprint30开发冻结基线。
## 1.2 四大维度评审总评
1. **TLMS整体架构**：三系统（IFFM/OMS/TLMS）通过通用reference表松耦合，结算域独立为Cost Domain，与执行域隔离，无职责交叉；
2. **欧洲物流业务真实性**：完整覆盖月度批量结算、Credit/Debit红字单据、一票多单分摊、承运商对账争议、快递/卡车双差异化费用模型，贴合荷兰欧盟财务监管要求；
3. **Odoo数据模型合理性**：采用Odoo原生Reference动态关联、ir.model业务对象绑定，无硬外键耦合，可扩展多业务单据；分摊、批次、争议Case分层分表，符合Odoo一对多标准设计；
4. **Sprint30实施风险**：P0仅落地基础数据底座与人工作业，自动匹配、AI OCR后置，迭代边界收敛，无范围蔓延，开发返工风险极低。
## 1.3 核心定位区分（明确Cost Domain与财务AP域边界）
1. **Carrier Settlement（TLMS成本域）**：业务对账层，负责承运商原始单据解析、业务匹配、成本分摊、差异争议处理、结算批次归集；**仅输出对账数据，不生成财务应付凭证**；
2. **财务AP应付域（独立Account模块）**：接收本模块已审批结算批次，自动生成Supplier Invoice/贷项凭证，负责付款、总账记账；
3. 二者数据流单向：Settlement Batch → 财务AP，双向修改隔离，避免业务对账与财务记账数据冲突。

# 2 全局分层系统架构（三系统松耦合）
```
上游业务系统
├─ IFFM 进口货代（到港通知、提单、柜、提货码）
└─ OMS 仓储订单管理（客户PO、派送单）
        ↓ 通用全局业务引用 transport.reference（Odoo Reference标准）
TLMS 运输物流系统
├─ Execution Domain 运输执行域
│   Transport Order / Transport Event / Transport Charge（系统预估成本）
├─ Cost Domain 成本核算域（本模块）
│   carrier.company 承运商主数据
│   carrier.charge.type 分级费用类型
│   carrier.match.rule 可配置匹配条件引擎
│   carrier.billing.document 承运原始单据
│   carrier.billing.line 单据明细
│   carrier.settlement.batch 月度结算批次
│   carrier.settlement.case 对账争议处理单
│   carrier.settlement.allocation 费用分摊记录
│   carrier.allocation_history 分摊变更审计日志
│   matching.history 匹配操作日志
└─ Reporting Domain 成本绩效报表域
        ↓ 已审批结算Batch推送
财务Account AP应付域
    供应商发票、Credit Note、付款台账、总账记账
```

# 3 业务痛点全版本演进汇总
## V1.0 初代缺陷
1. 混淆Invoice与结算单，不支持红字冲销；
2. 账单明细与运输订单一对一硬绑定，无法多单分摊；
3. 业务标识分散独立字段，新增编码需改表；
4. 无批量月度结算概念，财务无法按账期汇总；
5. 无争议异常处理载体，匹配差异人工线下记录。
## V1.1 中期改进遗留问题
1. transport.reference仅绑定Transport Order，无法关联IFFM/OMS单据；
2. allocation缺少费用类型维度，成本报表无法区分运费/附加费；
3. 无Settlement Batch月度批次，财务对账颗粒度不符；
4. 匹配仅权重无复合条件，多字段联合匹配准确率低；
5. 分摊修改无独立审计表，金额篡改无完整追溯。

# 4 业务场景与单据分类
## 4.1 运输模式（carrier.company分层区分）
1. TRUCK 整车/零担陆运：码头入仓、仓间调拨、空柜、跨境门到门
2. EXPRESS 国际快递：DHL/DPD/GLS/UPS派件退货
3. 3PL 综合第三方物流商
## 4.2 承运单据document_type枚举（欧盟财务标准）
| 单据类型 | 业务说明 | 金额符号规则 |
| --- | --- | --- |
| Invoice 税务发票 | 合规正式增值税发票 | positive |
| Statement 月度结算单 | 承运月度汇总费用清单 | positive |
| Debit_Note 借项单 | 补收漏计费用 | positive |
| Credit_Note 贷项单 | 扣款、退款冲抵 | negative |
| Cost_Sheet 简易费用清单 | 无税务效力临时明细 | positive |
## 4.3 核心业务对象新增说明
1. carrier.settlement.batch：月度结算批次，财务对账最小单元，一组同承运商、同账期全部billing.document；
2. carrier.settlement.case：对账争议工单，承载无法匹配、金额差异、承运商纠纷单据，专人认领闭环。

# 5 端到端完整业务流程
1. 承运商按月提供PDF/Excel单据，结算人员上传至系统；
2. 系统按承运商专属解析模板生成billing_document主单、billing_line明细；
3. 自动匹配引擎读取transport.reference全局业务索引，按配置复合条件匹配运输订单；
4. 系统自动按柜/票数比例生成allocation分摊记录；
5. 匹配失败、金额差异单据自动生成settlement.case争议工单分配负责人；
6. 结算专员人工调整分摊，所有金额变更写入allocation_history审计日志；
7. 同承运商、同账期单据归集至同一个settlement.batch；
8. 分层校验：文件解析校验→匹配分摊校验→差异复核；
9. 财务审批Batch，推送至Account AP模块生成应付凭证；
10 财务标记付款后批次正式归档，全流程操作永久留痕不可删除。

## 6.5 carrier.billing.document 承运单据主单
### 基础字段
document_id、carrier_id、document_no、document_type、document_date、billing_period_start/end、currency、total_amount、amount_sign、附件

1. 文件解析状态（仅代表上传解析）
`Draft → Uploaded → Parsed → Validated / Validation_Failed`

`Matching → Exception（争议Case绑定） → Review → Approved → Paid → Closed`

# 7 匹配引擎执行逻辑
1. 系统遍历当前billing_line内所有ref_value；
2. 按当前承运商match.rule配置顺序执行复合条件检索transport.reference；
3. 命中多条按priority_score取最高置信匹配结果；
4. 无有效匹配自动生成settlement.case争议工单，进入人工处理。

# 8 分摊与变更管控规则
1. 自动分摊：同单据明细按柜/订单数量比例拆分至对应运输单；
2. 人工分摊限制：仅Settlement Clerk可修改金额，**必填change_reason**；
3. 每次金额修改自动写入allocation_history，永久留存新旧金额对比；
4. Credit_Note红字单据分摊时自动扣减对应订单成本。

# 9 精细化三层角色权限（V1.2优化）
## 1. Logistics Operator 调度
可读全部单据、匹配、争议；仅可提交匹配建议，**禁止修改分摊金额**
## 2. Settlement Clerk 结算专员
完整匹配、人工分摊、处理争议Case；修改金额强制填写变更理由；无财务审批权限
## 3 Finance 财务人员
只读所有对账、分摊、审计记录；可审批结算批次、标记付款；禁止调整分摊数据
## 4 System Admin
仅维护费用类型、匹配规则、解析模板，无业务操作权限
