## 1. 系统架构总览

### 1.1 模块依赖关系

```
transport_logistics_management
  ├── base (核心)
  ├── mail (消息/日志)
  ├── stock (仓库)
  ├── account (财务)
  ├── portal (门户)
  ├── contacts / product (基础数据)
  ├── fleet (车队，可选)
  └── report (打印)
  
可选上游引用（无硬依赖）：
  ├── wd_iffm（进口货代提货需求，通过 Reference 字段引用）
  └── worlddepot（WMS 基础模型，间接引用）
```

### 1.2 层级结构

```
┌──────────────────────────────────────────────────┐
│                 菜单 / 导航层                       │
│   Pickup & Service / Quotation / Execution        │
└──────────────────┬───────────────────────────────┘
                   │
┌──────────────────▼───────────────────────────────┐
│                Controller 层                      │
│    PickupScheduleController (6 routes)           │
└──────────────────┬───────────────────────────────┘
                   │
┌──────────────────▼───────────────────────────────┐
│                Service / Model 层                  │
│  pickup.plan → transport.request → inquiry       │
│  → quote → order → tracking / pod / cmr         │
│  container.service.request → order               │
│  container.master (全局柜档案)                    │
└──────────────────┬───────────────────────────────┘
                   │
┌──────────────────▼───────────────────────────────┐
│                数据存储层                          │
│  PostgreSQL / Odoo ORM                           │
└──────────────────────────────────────────────────┘
```

### 1.3 流程二分架构

```
                    pickup.plan / container.service.request
                               │
                      destination_type / request_type
                               │
              ┌────────────────┴────────────────┐
              ▼                                  ▼
        计划驱动型 (场景1/5/8)             商务报价型 (场景2/3/4/6)
              │                                  │
              ▼                                  ▼
     pickup.plan.scheduled_date        pickup.plan → transport.request
              │                             (action_create_transport_request)
              │                                  │
              ▼                                  ▼
     tlmp.transport.order              tlmp.transport.inquiry (多轮询价)
              │                                  │
              │                           tlmp.transport.quote (加价)
              │                                  │
              │                                  ▼
              └──────────→ tlmp.transport.order ←┘
                                          │
                                          ▼
                              tlmp.transport.tracking
                              tlmp.pod + tlmp.cmr (签收/结算)
```

## 2. 数据模型设计

### 2.1 `pickup.plan` — 提货需求

**Odoo 标识**：`pickup.plan`

**用途**：统一的需求容器，支持柜和托件两种货物类型，是排期日历和询价报价流程的统一入口。

#### 字段表

| 字段名 | 类型 | 必填 | 默认值 | 说明 |
|--------|------|------|--------|------|
| `name` | Char | 是 | `_('New')` | 需求编号，由序列 `pickup.plan.seq` 自动生成，格式 `PP-YYMM-XXXX` |
| `cargo_type` | Selection | 是 | `container` | `container` / `pallet`，决定显示哪些明细字段 |
| `source_type` | Selection | 是 | `manual` | `iff` / `manual`，需求来源；iff 来源所有明细字段只读 |
| `iff_requirement_ref` | Reference | 否 | — | 可选，指向 `import.pickup.requirement`，无硬依赖 |
| `iff_plan_ref_id` | Integer | 否 | — | IFFM 单据内部 ID，用于上游生命周期联动锁定 |
| `destination_type` | Selection | 是 | `warehouse` | `warehouse` / `warehouse_transfer` / `customer` / `self_pickup` |
| `warehouse_id` | Many2one | 条件 | — | 目标仓库，destination_type = warehouse/warehouse_transfer 时必填 |
| `source_warehouse_id` | Many2one | 条件 | — | 发货仓库，destination_type = warehouse_transfer 时必填 |
| `is_bonded_transfer` | Boolean | — | False | 保税调拨标记；出/入仓任一方为保税仓时强制勾选 |
| `partner_id` | Many2one | 条件 | — | 客户，destination_type ≠ warehouse 时必填 |
| `delivery_address` | Text | 否 | — | 客户地址 |
| `delivery_contact` | Char | 否 | — | 客户联系人 |
| `delivery_phone` | Char | 否 | — | 客户联系电话 |
| `container_line_ids` | One2many | — | — | 集装箱行明细，cargo_type=container 时显示 |
| `pallet_count` | Integer | 否 | 0 | 托盘数 |
| `package_count` | Integer | 否 | 0 | 件数 |
| `cargo_weight` | Float | 否 | 0.0 | 重量 (kg) |
| `cargo_volume` | Float | 否 | 0.0 | 体积 (m³) |
| `cargo_description` | Text | 否 | — | 货物描述 |
| `carrier_id` | Many2one | 否 | — | 卡车公司，default 从系统参数 `tlmp.default_pickup_carrier_id` |
| `planned_pickup_date` | Datetime | 否 | — | 计划提货时间 |
| `driver_name` | Char | 否 | — | 司机 |
| `driver_phone` | Char | 否 | — | 司机电话 |
| `vehicle_plate` | Char | 否 | — | 车牌号 |
| `scheduled_date` | Date | 否 | — | 排期日期（日历拖拽后自动填充） |
| `transport_request_id` | Many2one | — | — | 关联 `tlmp.transport.request`（商务报价型流程入口） |
| `inquiry_id` | Many2one | — | — | 关联 `tlmp.transport.inquiry` |
| `transport_order_id` | Many2one | — | — | 关联 `tlmp.transport.order` |
| `notes` | Text | 否 | — | 备注 |
| `company_id` | Many2one | 是 | 当前公司 | 公司域 |

#### 状态说明

`pickup.plan` 没有独立状态字段，**不设状态机**。其生命周期由以下派生字段间接管理：
- `scheduled_date` → 有值即为已排期
- `transport_request_id` → 有值即为已进入询价报价链路
- `transport_order_id` → 有值即为已生成运输订单

#### 关键约束

| 约束 | 实现方式 |
|------|---------|
| IFFM 来源明细只读 | `write()` 方法拦截 `container_number/weight/type/bl_number/seal_number` 的修改 |
| 保税调拨强制勾选 | view 层 + server 层双重校验（条件：destination_type = warehouse_transfer 且任意仓库为保税仓） |
| IFFM 生命周期锁定 | `iff_plan_ref_id` 匹配上游状态，上游作废时自动置为只读 |

#### 序列

```xml
code: pickup.plan.seq
prefix: PP-YYMM-
padding: 4
```

### 2.3 `container.master` — 全局柜档案

**Odoo 标识**：`container.master`

**用途**：柜号唯一主数据，所有涉及柜号的外键统一关联此档案。一期仅做基础录入，二期扩展全生命周期管理。

#### 字段表

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `container_number` | Char | 是 | 柜号，唯一索引，自动转大写去空格 |
| `container_type` | Selection | 是 | 箱型 |
| `owner_id` | Many2one | 否 | 柜主（船公司/租赁公司） |
| `current_location_id` | Many2one | 否 | 当前位置 |
| `current_status` | Selection | 否 | 当前状态：loaded/empty/at_depot/with_carrier/damaged |
| `last_service_date` | Date | 否 | 最后维护日期 |
| `notes` | Text | 否 | 备注 |
| `active` | Boolean | 是 | 启用 |

#### 约束

- `container_number` 全局唯一
- `create()` 中自动 `.strip().upper()` 清洗柜号

### 2.4 `container.service.request` — 空柜调拨申请

**Odoo 标识**：`container.service.request`

**用途**：堆场/码头 ↔ 仓库之间的空柜调拨管理，独立于货载运输单据。

#### 字段表

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `name` | Char | 是 | 申请编号，序列 `container.service.request.seq` |
| `request_type` | Selection | 是 | `depot_to_warehouse` / `warehouse_to_depot` |
| `container_master_id` | Many2one | **是** | 关联 `container.master`，禁止手动输入文本柜号 |
| `display_container_number` | Char | — | related，柜号显示 |
| `display_container_type` | Selection | — | related，箱型显示 |
| `quantity` | Integer | 否 | 数量，默认 1 |
| `depot_location_id` | Many2one | 条件 | 堆场/码头 |
| `warehouse_id` | Many2one | 条件 | 仓库 |
| `carrier_id` | Many2one | 否 | 卡车公司 |
| `planned_date` | Date | 否 | 计划日期 |
| `driver_name` / `driver_phone` / `vehicle_plate` | Char | 否 | 司机信息 |
| `state` | Selection | 是 | `draft` → `confirmed` → `completed` / `cancelled` |
| `transport_order_id` | Many2one | — | 关联 `tlmp.transport.order` |
| `notes` | Text | 否 | 备注 |

#### 状态机

```
draft ──action_confirm──→ confirmed ──action_complete──→ completed
  │                          │
  └──action_cancel──────────→ cancelled
```

在 `confirmed` 状态下点击 `action_create_transport_order()` 生成运输订单：
- `request_type = depot_to_warehouse` → `transport_type = port_to_warehouse`
- `request_type = warehouse_to_depot` → `transport_type = to_customer`

## 3. 流程设计

### 3.1 场景 → 流程映射表

| 场景 | 流程类型 | 入口模型 | 排期日历 | 询价报价 | 运输订单 |
|------|---------|---------|---------|---------|---------|
| 1. Terminal → 我方仓库 | 计划驱动 | `pickup.plan` | ✅ | ❌ | ✅ (直接创建) |
| 2. Terminal → 客户地址 | 商务报价 | `pickup.plan` | ❌ | ✅ | ✅ (报价确认后) |
| 3. 仓库 → 客户地址 | 商务报价 | `pickup.plan` | ❌ | ✅ | ✅ |
| 4. 客户A → 客户B | 商务报价 | `pickup.plan` | ❌ | ✅ | ✅ |
| 5. 仓库A → 仓库B (调拨) | 计划驱动 | `pickup.plan` | ✅ | ❌ | ✅ |
| 6. 客户 → 仓库 (退货) | 商务报价 | `pickup.plan` | ❌ | ✅ | ✅ |
| 7. 柜到仓换柜 | 柜级标记 | `tlmp.transport.container` | — | — | 字段扩展 |
| 8. Depot ↔ 仓库 (空柜) | 计划驱动 | `container.service.request` | ❌ | ❌ | ✅ (直接创建) |

### 3.2 计划驱动型流程（场景 1/5）

```
pickup.plan 创建
  │
  ▼
destination_type = warehouse / warehouse_transfer
  │
  ├── 打开排期日历 (Schedule page)
  │
  ├── 左侧列表 → 拖拽柜/托件到日期 → scheduled_date 写入
  │
  └── action_create_transport_order (或在日历页触发)
        │
        ▼
  tlmp.transport.order.create({
    transport_type → terminal_to_warehouse / warehouse_transfer
    carrier_id → 从 plan.carrier_id
    container → 从 plan.container_line_ids
    cargo → 从 plan.pallet_count / cargo_weight
  })
```

**条件校验**：
- `is_bonded_transfer = True` 时，出/入库任一方为保税仓则强制勾选
- 调拨场景 `source_warehouse_id` 和 `warehouse_id` 均必填且不能相同

### 3.3 商务报价型流程（场景 2/3/4/6）

```
pickup.plan 创建 (destination_type = customer / self_pickup)
  │
  ▼
action_create_transport_request()
  │
  ▼
tlmp.transport.request 创建
  ├── partner_id = plan.partner_id
  ├── transport_type → 映射
  ├── cargo_description → 从 pallet/container 字段
  └── 打开表单让用户补充
  │
  ▼
tlmp.transport.inquiry (向承运商询价)
  ├── 可多轮：sent → responded → accepted / rejected
  └── rejected 后支持重新 send（已实现）
  │
  ▼
tlmp.transport.quote (我方加价)
  ├── draft → sent → accepted / rejected / cancelled
  ├── accepted → 自动创建 transport order
  ├── rejected → 回退 inquiry 至 sent 状态（可重新询价）
  └── cancelled → 归档，不可再创建订单
  │
  ▼
tlmp.transport.order
```

### 3.4 空柜调拨流程（场景 8）

```
container.service.request 创建
  │
  ▼
action_confirm() → state = confirmed
  │
  ▼
action_create_transport_order()
  │
  ▼
tlmp.transport.order 创建
  ├── container_number.from container_master_id
  ├── transport_type 根据 request_type 映射
  └── transport_order_id 写回 request
```

## 4. 前端设计

### 4.1 排期日历（Schedule Calendar）

**路由**：`/pickup_schedule`

**页面布局**：

```
┌─────────────────────────────────────────────────────────────┐
│  [[Month Nav]]                    Destination: [▼]  [▼]   │
├──────────────────┬──────────────────────────────────────────┤
│  Containers      │  Calendar (按天网格)                    │
│  ──────────      │  ┌───┬───┬───┬───┬───┬───┬───┐         │
│  Destination: ▼  │  │Mo │Tu │We │Th │Fr │Sa │Su │         │
│  Warehouse:   ▼  │  ├───┼───┼───┼───┼───┼───┼───┤         │
│                  │  │ 1 │ 2 │ 3 │ 4 │ 5 │ 6 │ 7 │         │
│  MSCU1234567     │  │ 2│   │   │ 1│   │   │   │         │
│  [W][Rotterdam]  │  ├───┼───┼───┼───┼───┼───┼───┤         │
│  BL: NKGA84065   │  │ 8 │ 9 │10 │11 │12 │13 │14 │         │
│                  │  │   │   │   │   │   │   │   │         │
│  Pallet Job-123  │  └───┴───┴───┴───┴───┴───┴───┘         │
│  [Pallet] 12pl   │                                          │
│  Wt: 2400 kg     │                                          │
└──────────────────┴──────────────────────────────────────────┘
```

**交互逻辑**：
- 左侧容器/托件卡片 `draggable="true"`
- 日历格子 `dropzone` 监听，阻止默认行为
- `dragstart` → `e.dataTransfer.setData('text/plain', planId)`
- `drop` → 调用 `/api/schedule` → 设置 `pickup.plan.scheduled_date`
- 点击 × 按钮 → 调用 `/api/unschedule` → 清空 `pickup.plan.scheduled_date`
- 日历格内显示已排期柜/托件的柜号或需求名，可点击 × 移除
- 当日高亮蓝色

**筛选逻辑**：
- 目的地筛选（All / Warehouse / Customer / Self-Pickup）
- 仓库筛选（destination = warehouse 时展开，选项来自 `stock.warehouse`）

**API 数据流**：
```
loadContainers()
  ├── container.line: search_read([]) → 所有集装箱行
  ├── pallet plans: search_read([cargo_type=pallet]) → 所有托件需求
  └── all plans: search_read([]) → 用于目的地+仓库过滤

renderCalendar()
  └── /api/schedules?year=&month= → 已排期的 plans → 按 date 分组渲染
```

### 4.2 标准表单视图

所有模型使用 Odoo 标准 `form/list/search` 视图模式，遵循以下布局惯例：

1. **Header**: state buttons (visible/invisible based on state) + statusbar
2. **Sheet top**: 分组两列布局，基本信息左列，业务信息右列
3. **Notebook**: 明细行、关联单据、备注等分页
4. **Chatter**: 消息和日志

菜单结构见 `6. 系统菜单`。

## 5. 控制器 API 设计

### 5.1 路由总表

| 路由 | 类型 | 方法 | 用途 |
|------|------|------|------|
| `/pickup_schedule` | http (website) | GET | 排期日历页面 |
| `/pickup_schedule/api/schedule` | json | POST | 设置排期日期 |
| `/pickup_schedule/api/unschedule` | json | POST | 取消排期 |
| `/pickup_schedule/api/schedules` | json | GET | 获取排期列表 |

### 5.2 `POST /pickup_schedule/api/schedule`

**请求**：
```json
{
  "plan_id": 123,
  "scheduled_date": "2026-07-15"
}
```

**处理逻辑**：
```python
plan = pickup.plan.browse(plan_id)
plan.scheduled_date = scheduled_date
return {'ok': True, 'day_count': count_of_plans_on_date}
```

### 5.3 `POST /pickup_schedule/api/unschedule`

**请求**：
```json
{
  "plan_id": 123
}
```

**处理逻辑**：
```python
plan = pickup.plan.browse(plan_id)
the_date = plan.scheduled_date
plan.scheduled_date = False
return {'ok': True, 'day_count': count_of_remaining_plans_on_date}
```

### 5.4 `GET /pickup_schedule/api/schedules`

**参数**：`year`, `month`（可选，用于按月过滤）

**响应**：
```json
[
  {
    "plan_id": 123,
    "scheduled_date": "2026-07-15",
    "container_number": "MSCU1234567",
    "container_type": "40HQ",
    "bl_number": "NKGA84065",
    "item_type": "container"
  },
  {
    "plan_id": 124,
    "scheduled_date": "2026-07-15",
    "container_number": "PP-2607-0001",
    "pallet_count": 12,
    "item_type": "pallet"
  }
]
```

**处理逻辑**：
```python
plans = pickup.plan.search([('scheduled_date', '>=', start),
                            ('scheduled_date', '<', end)])
for p in plans:
    if p.cargo_type == 'pallet':
        # 添加一个条目
    else:
        # 遍历 container_line_ids，每个柜一条
```

## 8. 跨模块集成设计

### 8.1 与 IFFM 集成

| 方向 | 技术实现 |
|------|---------|
| 引用方式 | `Reference` 字段，可选 `import.pickup.requirement`，无 `depends` |
| 数据同步 | 需求创建时 `onchange` 拉取明细，之后独立，不做实时同步 |
| 只读约束 | `source_type='iff'` 时 `write()` 拦截明细修改 |
| 生命周期 | 上游作废时 `iff_plan_ref_id` 标记锁定 TMS 记录 |
| **禁止** | TMS 绝不反向修改 IFFM 任何字段 |

### 8.2 与 wd_bonded_wms 集成

| 方向 | 技术实现 |
|------|---------|
| 信号方式 | `is_bonded_transfer = True` 触发信号（一期只做标记，二期实现联动） |
| 库存归属 | TMS 只做运输轨迹留痕，不做任何库存变动 |
| 账册核销 | B3/T1 申报全部由 WMS 模块执行 |

## 9. 关键约束与校验规则

### 9.1 写保护规则

```python
# pickup.plan.write() 拦截逻辑
IFF_PROTECTED = ['container_number', 'container_type', 'weight',
                 'bl_number', 'seal_number']

if rec.source_type == 'iff' and any(k in vals for k in IFF_PROTECTED):
    raise UserError('Cannot modify IFFM-sourced container details.')
```

### 9.2 Quote → Inquiry 回退

```python
# tlmp.transport.quote.action_reject()
if self.inquiry_id and self.inquiry_id.state == 'accepted':
    self.inquiry_id.write({'state': 'sent', 'response_date': False})
```

### 9.3 POD 完成约束

```python
# tlmp.transport.order.action_close()
if not self.pod_id or self.pod_id.state != 'confirmed':
    raise UserError('Cannot close: POD must be confirmed first.')
```

### 9.4 Quote 生命周期

```
draft → sent → accepted → (auto create order)
draft → sent → rejected → (return inquiry to sent)
draft → sent → cancelled → (archived, no order)
draft → sent → expired   → (auto cron)
```
